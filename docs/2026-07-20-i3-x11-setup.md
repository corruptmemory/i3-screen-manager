**Why it worked where the stacking WMs did not.** PekWM and FVWM3 both died the
same death: not a missing feature, but an accumulation of window-placement
behaviours you had to hold in your head. i3's tiling model is the cure, not the
compromise it was feared to be — windows have deterministic places, so the WM
never guesses and neither does the user. The whole detour existed to *avoid*
tiling; tiling turned out to be the thing that makes "most things just work."

## 1. The one that could have cost the most: `exec` vs `exec_always`

The shipped docs do not cover this (`man 5 i3` has no `exec_always` entry;
`/usr/share/doc/i3/` is images + a refcard). It was **measured** in a nested
Xephyr, both `i3-msg` calls confirming `{"success":true}`:

| | `exec` | `exec_always` |
| --- | --- | --- |
| startup | runs | runs |
| **reload** (`$mod+Shift+c`) | no | **no** |
| **restart** (`$mod+Shift+r`) | no | **yes** |

**This corrects the widespread belief that `exec_always` re-runs on *reload*.**
It re-runs on **restart**. Either way `exec` never re-runs, so the FVWM3
duplicate-daemon failure — daemons on `StartFunction`, which fires on every
restart — cannot happen with `exec`. Daemons use `exec`; only `feh` (which exits
immediately) uses `exec_always`.

The bigger point: i3 has `i3 -C`, a config check that never starts a session.
FVWM3 had no equivalent — its closest analogue was a Xephyr parse check that
turned out nearly worthless because `Read` failures were silent. `i3 -C` was
used after every edit here and caught mistakes before any boot.

## 2. Window-identity gotchas (the ones that bite)

Diagnosed by measurement, not guessed — the recurring theme of the whole i3
build.

- **Electron apps present a lowercase-`res_class` main window and
  capital-cased tray/helper windows.** Discord's real window is `discord`,
  Slack's is `slack`; their helpers are `Discord` / `Slack`. A first attempt at
  `class="^Slack$"` left Slack unmatched and loose on the focused workspace. Fix:
  match case-insensitively (`(?i)`). Keybase's main window is capital `Keybase`
  and matched directly.
- **Brave PWAs share `res_class` `Brave-origin`.** Messages and WhatsApp are
  distinguishable only by `instance` (`crx_<app-id>`). Matching by class would
  make them interchangeable.
- **Discord spawns an updater window first** (`title "Discord Updater"`).
  Anchoring the match to a title ending in `Discord` keeps the updater out of
  the slot. Verified on a cold start.

All three recur identically in the swallow rules (§3) and the `Super+F1..F5`
focus binds — same criteria, so focus and layout can never disagree about what
an app is.

## 3. The standout: the comms stack on workspace 10

The biggest win of the build, and the clearest demonstration that i3's tree is
predictable enough to *describe and trust*.

At login, workspace 10 (the portrait side monitor) builds itself: **Messages +
WhatsApp** tabbed on top, **Discord + Slack + Keybase** tabbed on bottom. It is
i3's own layout-restore, not a scripted mover:

- `~/.config/i3/chat-layout.json` — a `splitv` of two tabbed groups whose five
  leaves are empty **placeholders**, each with a `swallow` criterion.
- `~/.local/bin/i3-chat-layout` — `append_layout` the skeleton onto ws 10, then
  launch the five apps; i3 swallows each window into its slot.
- one `exec` line in `config-desktop` (not `exec_always` — see §1).

**Reachability:** `Super+F1..F5` = Messages / WhatsApp / Discord / Keybase /
Slack, focusing by criteria regardless of workspace.

**Safety net:** `~/.local/bin/i3-chat-rebuild` + a "Launch Chats" `.desktop`
entry (rofi drun / app menu) rebuild ws 10 from scratch. It must be a separate
script from the login launcher: swallowing only catches windows created *after*
the skeleton, and all five apps are single-instance, so re-running the login
launcher merely focuses the open windows. The rebuild forces a clean slate —
close windows, force-kill the survivors (Electron ignores graceful closes),
clear ws 10 — then hands off. Keybase's *service* daemon (KBFS) is left running;
only the GUI is killed.

Verified end to end: from a populated ws 10, one rebuild tore everything to zero
and restored all five to their correct slots in ~10s, updater and all, with no
duplicate windows even when a CLI run and a rofi launch overlapped.

## 4. Fixed / decided by checking reality, not the old config

- **Brightness binds removed.** `/sys/class/backlight/` is empty on this
  desktop and `brightnessctl` is not installed; the desktop IceWM config binds
  zero brightness keys while the laptop binds four. Brightness is laptop-only.
- **`dex` autostart removed.** Not in the Artix repos, AUR off-limits; every
  daemon is launched explicitly instead.
- **Both systemd lines removed.** `systemctl --user import-environment` is
  systemd-only (dead on OpenRC); `dbus-update-activation-environment` is
  required but its `--systemd` flag is not, so it moved to `.xinitrc-i3` without
  it.
- **`nm-applet` removed (post-first-day polish).** It was auto-started and
  parking a dead tray icon — because **NetworkManager is not the active network
  manager on this Artix box** (it is not running; the wired network is handled
  another way). The applet had nothing to talk to. Dropped from autostart.
  udiskie and flameshot stay — both earn their tray slot (flameshot must stay
  resident to back the `Print` keybind).
- **Speaker + mic added to the main bar (polish).** `internal/pulseaudio` (over
  pipewire-pulse) for the sink, `pulseaudio-control --node-type input` for the
  source; both scroll-to-adjust, click-to-mute. The mic `--format` is pinned to
  `'$VOL_ICON $VOL_LEVEL%'` because the default also prints the node name, which
  showed a bare `Source #NN` — the node-nicknames in the shared `config.ini` are
  the *laptop's* mic/headset and do not match this desktop's hardware.

## 5. Config derived from another config silently loses what you don't port

Two flavours of the same failure, hit **four** times in one build. Both fail
*silently* — no error, just missing behaviour.

**(a) Copy-deployment drift** — `~/.config/<x>` that is a real copy (not a
symlink into the repo) drifts ahead of git:

- **`~/.config/i3/config`** — live had `feh` wallpaper, a `$browser` var, and two
  binds the repo lacked (repo still had `waypaper`, a Wayland tool).
- **`~/.config/rofi/config.rasi`** — live `font: "TX-02 11"`, repo `12`.
- (earlier the same day) **polybar's `cmos-battery` module.**

**(b) Derive omission** — deriving a new config from an old one silently drops
whatever you don't consciously carry over:

- **`config-i3.ini` lost polybar's emoji `font-2`** (`JoyPixels`), so the newly
  added speaker/mic modules *loaded and worked* but polybar was `Dropping
  unmatched character '🔉'` — the icons rendered blank. The modules were fine;
  the font that renders their glyphs was missing. Same shape as the Ghostty
  theme that "never applied."

Fix for (a): propagate live → repo *before* editing, then symlink so it cannot
recur — **any `~/.config` path that is a plain copy is latent drift.** Fix for
(b): when a config says "derived from X", diff it against X and account for
every dropped line, and **watch the load log** — polybar's dropped-glyph warning
is only visible there, which is why a clean kill-and-relaunch (not
`polybar-msg cmd restart`) is what surfaced it.
