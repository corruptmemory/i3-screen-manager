# FVWM3 on `godlike-artix` — setup outcome

FVWM3 1.1.5 as a third, parallel session on the desktop, alongside IceWM (X11)
and Hyprland (Wayland). Built and debugged 2026-07-20.

**Status: RUNNING, in evaluation.** Every reported defect is fixed and the
session is usable, but Jim's verdict as of end-of-day is *"either residual jank
or my head hasn't fully adapted to the new workflow"* — so it is not yet
declared the daily X11 driver. IceWM remains untouched and `start-icewm` is
always one command away.

Design and rationale: `docs/2026-07-20-fvwm3-desktop-setup-plan.md`
Step-by-step build: `docs/2026-07-20-fvwm3-implementation-steps.md`

**Why FVWM3 at all:** it is the only stacking X11 window manager with genuinely
independent per-monitor workspaces (`DesktopConfiguration per-monitor`). See the
design doc for the full scorecard; Window Maker, CWM, ctwm, fluxbox, openbox,
pekwm, jwm, sawfish, xfwm4, marco and IceWM were all verified global.

## 1. The one that cost the most: `FVWM_USERDIR`

**fvwm3's user directory is `~/.fvwm`, NOT `~/.fvwm3`** — despite the binary
being named `fvwm3`. From `fvwm3(1)`:

> `-f config-file`  Causes fvwm to read config-file instead of **`~/.fvwm/config`**
> as its initialization file. `$FVWM_USERDIR` can also be used to change location
> of default user directory **`~/.fvwm`**.

The entire config was built around `~/.fvwm3` on an unchecked assumption. First
boot came up as stock 1992 FVWM — four desks, title bars, built-in pager.

That single fact caused **two** failures, and the second is the nastier:

1. `exec fvwm3` never located `~/.fvwm3/config`; it silently fell back to
   `/usr/share/fvwm3/default-config/config` and created an empty `~/.fvwm/`.
2. Even *with* `-f`, every `Read $[FVWM_USERDIR]/{styles,bindings,modules,autostart}`
   resolves into that empty `~/.fvwm/`. **Those `Read` failures are silent**, so
   all four sub-files would have loaded as nothing.

Fix: `export FVWM_USERDIR="$HOME/.fvwm3"` in `.xinitrc-fvwm3` before `exec
fvwm3`. One variable repairs both halves.

**Process lesson:** a Xephyr "parse check passed, zero errors" had been claimed
before this. It was worthless — it ran `fvwm3 -f ~/.fvwm3/config`, which parsed
the top-level file only, while the four `Read`s failed silently and produced
nothing to notice. *A config that loads is not a config that loaded everything.*

## 2. Every other defect, with root cause

Ordered as encountered. All were diagnosed by measurement (`xprop`, `xdotool`,
`ps`) rather than guessed.

| Symptom | Root cause | Fix |
| --- | --- | --- |
| Pager showed **one** workspace | (a) desks never declared — `_NET_NUMBER_OF_DESKTOPS = 1`; (b) `Module FvwmPager <alias>` with no desk range shows only the current desk | `DesktopName 0..9`; `Module FvwmPager <alias> 0 9` |
| `CMOS command not found` | `~/.local/bin/i3-cmos-battery` symlink missing (its sibling `aur-malware-check` was linked; this one was overlooked) | symlinked; reports 3.29V |
| Move/resize showed rubber-band outlines | fvwm defaults: `OpaqueMoveSize 5` means only windows under 5% of screen move solid; resize is a *separate* mechanism | `OpaqueMoveSize unlimited` + `Style * ResizeOpaque` |
| **Dead band on DP-2** — draggable into, not maximisable into | polybar advertises `_NET_WM_STRUT_PARTIAL top=268`, correct per EWMH (screen coordinates). **fvwm applies it as monitor-relative** and double-counts DP-2's y=240 offset → 508. Measured: maximised window at y=510. HDMI-1 at y=0 unaffected | polybar `override-redirect = true` (no strut) + `EwmhBaseStruts screen DP-2 0 0 28 0` |
| SloppyFocus unworkable | wrong model — focus tracking the pointer is trivially triggered by accident, and focus-without-raise means typing into a window you cannot see | `ClickToFocus, ClickToFocusRaises` |
| Pager vanished off desk 0 | it was a plain window on one desk (`_NET_WM_DESKTOP=0`, no sticky state) | `Style FvwmPager* Sticky, StaysOnTop, NeverFocus, …` |
| Windows covered the bar | **`override-redirect` does NOT imply always-on-top** — it only removes the bar from the *WM's* stacking control; plain X still stacks new windows above | polybar `wm-restack = generic` |
| Alt+Tab crossed monitors | used `CurrentScreen`, which despite its name is a synonym for `CurrentPageAnyDesk` and is **not monitor-scoped** | `Screen $[monitor.current]` |
| `Super+Shift+←/→` didn't follow the window | `MoveToDesk` alone sends and leaves you behind | `MoveToDesk` then `GotoDesk`, same relative arg |
| Mysterious "cross cursor" | fvwm's **window picker** — *"If no window was selected … the user is asked to select a window"* — triggered because nothing was focused | `Silent` prefix on all 17 window-targeting binds |
| No focus after sending a window away / after changing desk | fvwm under `ClickToFocus` never promotes a window to focus | `FocusHere` + `FvwmEvent new_desk` hook |
| Reloads spawned duplicate daemons | `StartFunction` runs on startup **and every restart** | daemons moved to `InitFunction`; modules stay in `StartFunction` |

### Two that are worth internalising

**`CurrentScreen` is a trap.** It reads like "the current monitor" and is
documented as *"windows that are on the current page of any desk"* — a synonym
for `CurrentPageAnyDesk`. The monitor-scoped condition is `Screen [name]`. This
was used in `SwitchMonitor` initially, meaning the "focus the destination
monitor" step could focus a window on the monitor you just left — the exact bug
the function existed to prevent.

**`StartFunction` vs `InitFunction`.** `StartFunction` runs on restarts too, so
long-running daemons hung off it multiply on every reload. Measured after a few
`Super+Shift+R`: polybar running 4 processes (both bars twice), udiskie twice.
`dunst` and `flameshot` self-deduped via single-instance logic — precisely the
luck that hides a bug. Modules are the opposite case and *must* stay in
`StartFunction`, since fvwm kills them on restart.

## 3. Deliberate design decisions

- **Bar is a hybrid**: FvwmPager (workspaces, `MiniIcons` for the app-icon look)
  + Polybar (title, stats, clock, tray). Only workspaces need per-monitor
  awareness; the rest already worked in `config.ini`.
- **No workspace module in Polybar, ever.** EWMH exposes a single global
  `_NET_CURRENT_DESKTOP`, so an EWMH workspace module can only show one
  monitor's state on both bars. There is a `grep` guard for this in the plan.
- **No window-title module on the portrait bar.** Same root limitation: one
  keyboard focus, one global `_NET_ACTIVE_WINDOW`, so it mirrored the other bar.
- **Number keys send-and-stay, arrow keys send-and-follow.** `Super+Shift+1..0`
  matches the existing IceWM `icesh setWorkspace` behaviour; `Super+Shift+←/→`
  follows. Asymmetric on purpose, documented in the keymap.
- **Focus fallback is an EVENT, not a keybind patch.** `FvwmEvent new_desk`
  catches every route to a desk change — keybinds, pager clicks, future scripts.
  A keybind-only fix would have left the pager path broken.

## 4. Xephyr cannot validate this project

Worth writing down because "just test it in Xephyr" is the obvious suggestion
and it does not work here.

Xephyr synthesises a **single RandR output** named `default`, and fvwm's
per-monitor mode is explicitly RandR-based. Measured:

| Attempt | Result |
| --- | --- |
| `-screen A -screen B +xinerama` | root becomes 2000x800, RandR still reports **one** output |
| `xrandr --setmonitor` (server reports RandR **1.6**, so supported) | silently no-ops |
| `xrandr --delmonitor default` | `BadValue` — automatic monitors cannot be deleted |

So per-monitor desks, the per-monitor pagers and the monitor warp — everything
carrying real risk — cannot be exercised in a nested server.

Xephyr *is* still good for one thing: a **config parse check**, which is cheap
insurance against a typo costing a session restart:

```bash
Xephyr -screen 1400x900 :6 >/dev/null 2>&1 &
sleep 4
timeout 10 env DISPLAY=:6 FVWM_USERDIR="$HOME/.fvwm3" fvwm3 2>&1 \
  | grep -iE "error|unknown|invalid"
```

Expect only `polybar ... Monitor "DP-2" not found` and an already-running polkit
agent — both are artefacts of the nested display, not config errors.

**Also note:** launch only dumb X clients (`xterm`, `xeyes`) inside Xephyr.
`ghostty`, `brave-origin` and `emacs` are all single-instance and hand off to
their live `:0` instance, opening windows on the *real* desktop.

## 5. Known gaps / backlog

- **`Super+Tab` window switcher is not ported.** It currently runs bare
  `rofi -show window`. IceWM has a bespoke `~/.local/bin/icewm-window-switcher`
  written precisely because `rofi -show window` activates via an EWMH
  `_NET_ACTIVE_WINDOW` message and leaves raising to the WM. That script drives
  IceWM through `icesh clients` / `icesh goto` / `icesh -window raise` +
  `xdotool windowfocus` — **entirely IceWM-specific IPC**. The FVWM3 equivalent
  needs rewriting against `FvwmCommand`/`FvwmMFL` or fvwm's own `WindowList`.
  Until then, expect the same half-working activation the IceWM script was
  written to fix. **This is the largest outstanding item.**
- **Alt+Tab is single-step**, not hold-Alt-and-cycle-an-MRU-list. Press
  repeatedly to walk the stack. Proper MRU needs `WindowList` or a module.
- **`Resize direction automatic`** computes no direction from the *exact* centre
  of a window (documented behaviour). Nudge toward the side you want.
- **`override-redirect` bars are unmanaged**, so they cannot be covered by a
  fullscreen window. If that becomes annoying, the alternative is aligning
  monitor *tops* (DP-2 at y=0), which sidesteps fvwm's strut bug entirely but
  gives up vertical centring.
- **No root menu.** The keymap cheat sheet lives as a header comment in
  `~/.fvwm3/bindings`; there is no in-session way to discover a binding.
- **Laptop not done.** The config hardcodes `DP-2`/`HDMI-1`, so `git pull` gives
  `nomad-artix` nothing useful. It would need a `.fvwm3-laptop/` variant the way
  `.icewm-laptop/` mirrors `.icewm/`.

## 7. Rollback

1. **Just run `start-icewm`.** IceWM and Hyprland were never modified.
2. Unlink: `rm ~/.fvwm3 ~/.xinitrc-fvwm3 ~/.local/bin/start-fvwm3` (all symlinks).
3. Uninstall: `sudo pacman -Rsp fvwm3` to DRY-RUN first, inspect, then `-Rs`.
